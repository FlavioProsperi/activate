Multiple VMs (v1.2)

Activate uses memory efficiently by maintaining soft references for the entities that are loaded from the storage.
To use this approach in an environment with multiple virtual machines using the same storage, there are two coordination mechanisms that transforms Activate into a Distributed STM:

Optimistic Offline Locking
This is a widely known pattern to deal with concurrency. It uses an entity version information to detect conflicts using the underlying storage.

Coordinator
A central service that synchronizes the in-memory data between VMs instances. At commit time, the coordinator receives all reads and writes of the transaction to detect conflicts.

The propose of these two mechanisms is to detect transaction conflicts at commit time (optimistic locking), then Activate can reload the entity information and retry the transaction if it is possible.

For a detailed explanation of the coordinator and offline locking internals, please refer to the Arquitecture Documentation.

The choice

When I should use the coordinator or offline locking?

The coordinator creates a “single point of failure” and probably it will be deprecated for the 1.3 version. It should be used only if your application can afford the risk of a coordinator failure.

Optimistic Offline Locking

There are two system properties to configure the offline locking:

activate.offlineLocking.enable
If this parameter is true, the offline locking is enabled. Default: false.

activate.offlineLocking.validateReads
If this parameter is true, the offline locking also validate all transactions reads against the storage version information. Default: false.

IMPORTANT: It is necessary to create the version column for all entities. It must be a Long value. Example migration:


class ExampleMigration extends Migration {
	def timestamp = 201208191834l
	def up = {
		table[MyEntity]
			.addColumn(
				_.column[Long]("version"))
	}
}

Coordinator

SERVER

Turn one of your application persistence context instance in a coordinator service just adding this system property:


activate.coordinator.server=true

An easy way is to add the -Dactivate.coordinator.server=true option to the application execution line.

CLIENTS

The other virtual machine instances must be clients. Set this system property:


activate.coordinator.serverHost=SERVER_HOST

Replace “SERVER_HOST” with the server context address.

SYNC THREAD

Coordinator guarantees that all transaction are consistent and entities are reloaded from the storage when needed.

To antecipate entities reload and avoid reload during transactions, there is a sync thread that reloads the necessary entities in a bach way.

PROPERTIES

System properties to customize coordinator:

activate.coordinator.port
Determines the coordinator server port. Default: 5674.

activate.coordinator.timeout
Time to wait coordinator server response in milliseconds. Default: 60000.

activate.coordinator.syncSleep
Sync thread sleep between batch reload. Default: 500.
